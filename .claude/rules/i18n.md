# i18n

next-intl v4 with `localePrefix: "never"` — locale is **not** part of the URL. Supported locales: `en`, `fr`.

## Locale resolution

`src/i18n/request.ts` runs on every SSR request:
1. Calls `auth()` to get the session
2. If authenticated, fetches the user's `locale` field from the DB
3. Falls back to `"en"` for unauthenticated requests or unknown locale values
4. Returns `{ locale, messages }` to next-intl

The locale in the JWT (`session.user.locale`) is **ignored** for UI resolution — it can be stale. Reading from DB ensures the UI updates immediately after the user changes their locale without requiring a re-login.

## Triggering a locale update

When `ProfileForm` saves a new locale, it calls `router.refresh()` afterward. This triggers a new SSR cycle which re-runs `request.ts`, re-reads the locale from the DB, and re-renders all server components with the correct language.

## Message files

`messages/en.json` and `messages/fr.json`. Keys are namespaced by feature. Current namespaces:

| Namespace | Used by |
|---|---|
| `nav` | AppNav, UserMenu |
| `topics` | topic pages and components |
| `items` | item cards, modals, forms |
| `scoring` | ScoreControls |
| `comments` | CommentList, CommentForm |
| `settings` | all /settings/* pages and SettingsSidebar |
| `auth` | login, signup pages |
| `profile` | ProfileForm |

When adding a new user-facing string: add it to **both** `en.json` and `fr.json` under the appropriate namespace before using `useTranslations` or `getTranslations`.

## Usage patterns

In Server Components:
```typescript
import { getTranslations } from "next-intl/server";
const t = await getTranslations("settings");
```

In Client Components:
```typescript
import { useTranslations } from "next-intl";
const t = useTranslations("items");
```

## Routing config

`src/i18n/routing.ts` defines locales and `localePrefix: "never"`. The `Locale` type is exported from this file and used throughout.
